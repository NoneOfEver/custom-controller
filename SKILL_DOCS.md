# SKILL — 项目文档深度增强（Rustdoc / Doxygen / Python）

> 目的：把“如何给项目补充高质量可维护文档”的做法固化成一份可复用的 SKILL。
> 适用：你希望在**不改变行为**或尽量少改代码的情况下，通过文档让后续维护者（尤其是跨语言背景）更快理解工程。
> 输出风格：优先使用“文档级注释”（Rustdoc / Doxygen / docstring），其次才用行内 `//` 解释语法点。

---

## 0. 触发条件（什么时候应该用这个 SKILL）

- 你准备交付/开源/交接一个工程，希望别人“先看文档就能懂 80%”。
- 你从 C/C++ 转 Rust（或反之），需要把关键概念用对方熟悉的术语解释清楚。
- 你希望把“硬件约束 / 协议格式 / 数据流 / 并发模型 / 生命周期”写成可审阅的文字。

非目标：
- 不是写营销文案。
- 不是把每一行都注释一遍（避免噪音）。

---

## 1. 总体原则（通用）

### 1.1 文档分层

优先级从高到低：
1. **根文档**：`README` / `HARDWARE` / `ENGINEERING_PLAN` / `docs/*`
2. **模块级文档**：文件顶部（Rust `//!` / C++ `/** */` / Python 模块 docstring）
3. **API 级文档**：公开函数/类型（Rust `///` / Doxygen `/** */` / Python 函数 docstring）
4. **行内注释**：只解释“为什么这样做”或“语法点/陷阱”，不重复代码字面意思

### 1.2 写什么（信息密度模板）

对每个模块/组件，至少回答：
- **它是什么**：职责一句话
- **输入/输出**：关键数据结构与接口
- **约束**：硬件/协议/性能/实时性/内存
- **并发/线程安全**：谁在写、谁在读，如何同步
- **失败模式**：常见错误与日志点
- **Reading Guide**：建议阅读顺序（对初学者非常有效）

### 1.3 不写什么（防止过度注释）

- 不解释显而易见的语句（如 `i += 1`）。
- 不在注释里复制粘贴整段代码。
- 不写会过期的硬编码细节（除非它是约束/协议）。

## 2. Rust（Rustdoc）文档打法

### 2.1 语法与位置

- **模块级**：文件顶部用 `//! ...`（解释“这个模块是什么/为什么存在/如何使用”）。
- **条目级（item）**：对 `pub struct/enum/fn/trait` 用 `/// ...`。
- **不要**在函数体语句前用 `///`：Rustdoc 只对 item 生效，会触发 `unused_doc_comment` 警告；语句解释用 `//`。

### 2.2 Rustdoc 推荐结构（可直接套用）

模块顶部：
```rust
//! <模块一句话职责>
//!
//! ## 约束
//! - ...
//!
//! ## 数据流
//! - ...
//!
//! ## Beginner Notes
//! - ...（所有权/'static/async/Mutex 等）
//!
//! ## Reading Guide
//! - 先看 ...
```

公开函数：
```rust
/// <一句话>
///
/// # 参数
/// - ...
///
/// # 行为
/// - ...
///
/// # 前置条件
/// - ...
///
/// # 失败模式
/// - ...
pub fn ... { ... }
```

### 2.3 初学者常见难点解释点（建议优先覆盖）

- 所有权/移动：为什么 `init(peripherals::X, pin, dma)` 要“拿走”参数。
- `'static`：为什么任务/驱动类型里常见。
- `async/await`：为什么 `await` 不等于阻塞整个系统。
- `Mutex`/临界区：为什么要锁、锁的范围为何要短。
- 静态内存：DMA buffer 用 `StaticCell` 的原因。

### 2.4 验收

- `cargo build` / `cargo check` 无 warning。
- `rustdoc` 不要求生成文档站点，但 Rustdoc 注释应能独立读懂。

## 3. C++（Doxygen）文档打法

### 3.1 位置与标签

- 文件头：`/** @file ... */` 描述模块职责、约束、依赖。
- 类/结构体：`/** ... */` + `@brief`。
- 函数：`@param`、`@return`、`@note`、`@warning`。

### 3.2 Doxygen 推荐模板

文件级：
```cpp
/**
 * @file
 * @brief <模块一句话职责>
 *
 * @details
 * - 约束：...
 * - 数据流：...
 * - 并发：...
 *
 * @section reading_guide Reading Guide
 * - 先看 ...
 */
```

函数级：
```cpp
/**
 * @brief <一句话>
 * @param foo ...
 * @return ...
 * @note 前置条件：...
 * @warning 失败模式/边界：...
 */
```

### 3.3 初学者/跨语言解释点

- RAII 与资源所有权：对应 Rust 的 move/Drop。
- 线程安全：mutex/atomic 的责任边界。
- 协议与字节序：明确 little-endian、CRC、对齐与 padding。

### 3.4 验收

- `doxygen`（如项目配置了）能无明显 warning。
- 读头文件/接口即可理解模块用途与调用方式。

## 4. Python（docstring）文档打法

### 4.1 选择一种 docstring 风格并保持一致

推荐其一：
- Google style
- NumPy style
- reST/Sphinx

### 4.2 推荐模板（Google style）

模块级：
```python
"""<模块一句话职责>

Constraints:
    - ...

Reading guide:
    1) ...
"""
```

函数级：
```python
def f(x: int) -> int:
    """<一句话>

    Args:
        x: ...

    Returns:
        ...

    Raises:
        ValueError: ...
    """
```

### 4.3 初学者/跨语言解释点

- 类型提示（typing）：让接口边界更清晰。
- I/O 与副作用：清楚写出读写文件/网络/串口行为。
- 测试：为关键解析器/协议编码器加最小单元测试示例。

### 4.4 验收

- `python -m pydoc <module>` 输出可读。
- 如有：`ruff`/`pylint`/`mypy` 无明显问题。

## 5. “教程工程”增量策略（可复用到任何语言）

建议按三步走：
1. **入口文档**：README 增加“工程导览 + Reading Guide + Bring-up Checklist”。
2. **关键路径**：把最重要的 1～2 条数据流写成可跟踪：例如“采样→快照→编码→发送”。
3. **语法点最小解释**：只在新手最容易卡住的点加 `Beginner Notes`。

---

## 6. 最小检查清单（交付前）

- 文档中提到的文件/符号都存在且名称正确。
- 约束（引脚/端序/时序/阈值）与代码一致。
- 编译/运行命令在 README 中是可执行的。
- 不把敏感信息（密钥/串口日志含隐私）写进仓库。
